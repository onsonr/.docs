## A

<AccordionGroup>

<Accordion title="Accumulator (Zero Knowledge Proofs)">

A cryptographic primitive that allows for compact representation of a set of elements, enabling membership proofs without revealing the elements.

</Accordion>

<Accordion title="Air drops">

A distribution of free tokens to users' wallets by a crypto project, aiming to increase interest and adoption of their token.

</Accordion>

<Accordion title="Arbitrage">

The practice of profiting from price discrepancies across different markets by simultaneously buying and selling assets.

</Accordion>

</AccordionGroup>

## B

<AccordionGroup>

<Accordion title="Bonding">

The process of delegating tokens to a validator to earn staking rewards and obtain voting power in a proof-of-stake system. Also referred to as delegating or staking.

</Accordion>

<Accordion title="Burn">

The permanent removal and destruction of tokens from the circulating supply.

</Accordion>

</AccordionGroup>

## C

<AccordionGroup>

<Accordion title="CometBFT">

A fork of the Tendermint consensus engine with various performance and security improvements.

</Accordion>

<Accordion title="Commission">

The percentage of staking rewards that a validator retains before distributing the remainder to their delegators.

</Accordion>

<Accordion title="Community pool">

A reserve of tokens set aside for funding community-driven projects through the governance process.

</Accordion>

<Accordion title="CosmWasm">

A smart contract platform built for the Cosmos ecosystem, allowing developers to create multi-chain smart contracts.

</Accordion>

<Accordion title="Cosmos SDK">

A framework for building application-specific blockchains that can interoperate with other blockchains.

</Accordion>

</AccordionGroup>

## D

<AccordionGroup>

<Accordion title="dApp">
A decentralized application built on a blockchain or other distributed ledger technology.
</Accordion>

<Accordion title="DDoS">
A Distributed Denial of Service attack, where an attacker overwhelms a network with traffic to disrupt its normal operation.
</Accordion>

<Accordion title="Decentralized Identity">
An identity system where individuals have control over their personal data and how it's shared.
</Accordion>

<Accordion title="Decentralized Storage">
A system that stores data across a distributed network of nodes, ensuring data redundancy and availability.
</Accordion>

<Accordion title="DeFi">
Decentralized Finance, a movement aimed at creating open, transparent, and permissionless financial systems using blockchain technology.
</Accordion>

<Accordion title="Delegated Proof-of-Stake (DPoS)">
A consensus mechanism where token holders vote for validators to secure the network.
</Accordion>

<Accordion title="Delegator">
A user who delegates their tokens to a validator to earn staking rewards in a proof-of-stake system.
</Accordion>

<Accordion title="DID">
Decentralized Identifier, a globally unique and persistent identifier for a person, organization, or thing.
</Accordion>

<Accordion title="DID Controller">
An entity that has the capability to make changes to a DID Document, essentially having control over the DID.
</Accordion>

<Accordion title="DID Document">
A JSON-LD document containing metadata associated with a DID, such as public keys and service endpoints.
</Accordion>

<Accordion title="DID Method">
Specifies the syntax and procedures for specific DID schemes. It defines how to read, write and resolve DIDs for a particular blockchain or storage network.
</Accordion>

<Accordion title="DID Resolver">
A system function that takes a DID as input and produces a DID Document as output, essentially translating the identifier into a form that the system can process.
</Accordion>

</AccordionGroup>

## E

<AccordionGroup>

<Accordion title="Elliptic Curve Integrated Encryption Scheme (ECIES)">

A hybrid encryption scheme that uses elliptic curve cryptography for key agreement and symmetric encryption for data encapsulation.

</Accordion>

</AccordionGroup>

## G

<AccordionGroup>

<Accordion title="Gas">
A measure of the computational effort required to execute a transaction or smart contract operation on a blockchain network.
</Accordion>

<Accordion title="Governance proposal">
A proposal submitted by a community member to suggest changes or improvements to a decentralized protocol, often subject to a voting process.
</Accordion>

</AccordionGroup>

## H

<AccordionGroup>

<Accordion title="HTMX">
A library that allows you to access modern browser features directly from HTML, without using JavaScript.
</Accordion>

</AccordionGroup>

## I

<AccordionGroup>

<Accordion title="Impermanent Loss">
The temporary loss of funds liquidity providers may experience due to the volatility in a trading pair.
</Accordion>

<Accordion title="Interchain Accounts">
A module in the Cosmos SDK that enables an account on one blockchain to control an account on another blockchain.
</Accordion>

<Accordion title="Interblockchain Communication (IBC)">
A protocol that enables reliable, ordered, and authenticated communication between heterogeneous blockchains.
</Accordion>

</AccordionGroup>

## J

<AccordionGroup>

<Accordion title="Jailed">
A state where a validator is temporarily prevented from participating in consensus due to misbehavior or poor performance.
</Accordion>

<Accordion title="JSON Web Token (JWT)">
An open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object. JWTs can be signed using a secret (with the HMAC algorithm) or a public/private key pair using RSA or ECDSA, which verifies the integrity of the claims contained within the token.
</Accordion>

</AccordionGroup>

## K

<AccordionGroup>

<Accordion title="Keyshare">
A portion of a private key that is distributed among multiple parties in a secure multi-party computation scheme.
</Accordion>

</AccordionGroup>

## L

<AccordionGroup>

<Accordion title="Liquidity Mining">
An incentive scheme where users are rewarded with tokens for providing liquidity to a decentralized exchange or lending platform.
</Accordion>

<Accordion title="LP Tokens">
Tokens issued to users who deposit assets into a liquidity pool, representing their share of the total pool.
</Accordion>

</AccordionGroup>

## M

<AccordionGroup>

<Accordion title="Matrix Protocol">
An open standard for secure, decentralized, real-time communication, providing a P2P routing layer.
</Accordion>

<Accordion title="Multi-Party Computation (MPC)">
A cryptographic protocol that allows parties to jointly compute a function over their inputs while keeping them private.
</Accordion>

</AccordionGroup>

## P

<AccordionGroup>

<Accordion title="Passkeys">
Secure, anonymous WebAuthn-based credentials for privacy-preserving, user-friendly authentication.
</Accordion>

<Accordion title="Pinecone">
A P2P routing layer in the Matrix Protocol that enables efficient message routing between embedded clients.
</Accordion>

<Accordion title="Public Key Assertion (WebAuthn)">
**PublicKeyAssertion** in WebAuthn is cryptographic proof of private key possession to verify identity without passwords.
</Accordion>

<Accordion title="Public Key Attestation (WebAuthn)">
PublicKeyAttestation in WebAuthn verifies a credential's authenticity via an attestation statement from the authenticator.
</Accordion>

</AccordionGroup>

## R

<AccordionGroup>

<Accordion title="Relaying Party">
An entity that relies upon a verifiable credential or authenticated identity during a transaction.
</Accordion>

</AccordionGroup>

## S

<AccordionGroup>

<Accordion title="Self-Sovereign Identity">
A model where individuals have sole ownership and control over their digital identities.
</Accordion>

<Accordion title="Service Record">
A record on the Sonr blockchain that represents a service, such as a dApp, and its associated metadata.
</Accordion>

<Accordion title="Slashing">
A penalty mechanism in proof-of-stake systems where a validator's tokens are deducted or destroyed for misbehavior or rule violations.
</Accordion>

<Accordion title="Slippage">
The difference between the expected price of a trade and the actual execution price, often due to market volatility or low liquidity.
</Accordion>

<Accordion title="Sonr">
A decentralized identity and asset management platform built on the Cosmos blockchain.
</Accordion>

</AccordionGroup>

## T

<AccordionGroup>

<Accordion title="Tendermint">
A Byzantine Fault Tolerant (BFT) consensus engine used by the Cosmos SDK.
</Accordion>

<Accordion title="Testnet">
A parallel network used for testing and development purposes, where the tokens have no real value.
</Accordion>

</AccordionGroup>

## U

<AccordionGroup>

<Accordion title="Unbonding">
The process of withdrawing delegated tokens from a validator, which typically requires a waiting period before the tokens become freely transferable.
</Accordion>

<Accordion title="Undelegate">
The act of removing delegated tokens from a validator, initiating the unbonding process.
</Accordion>

<Accordion title="Uptime">
The percentage of time a validator is online and actively participating in consensus, often used as a measure of their reliability.
</Accordion>

</AccordionGroup>

## V

<AccordionGroup>

<Accordion title="Validator">
A node in a proof-of-stake blockchain responsible for validating transactions and adding new blocks.
</Accordion>

<Accordion title="Verifiable Credential">
A tamper-evident credential with authorship that can be cryptographically verified.
</Accordion>

</AccordionGroup>

## W

<AccordionGroup>

<Accordion title="WebAuthn">
A web standard for authenticating users using public-key cryptography instead of passwords.
</Accordion>

<Accordion title="Witness (Zero Knowledge Proofs)">
In a zero-knowledge accumulator, a value that allows a prover to demonstrate membership of an element without revealing the element itself.
</Accordion>

</AccordionGroup>

## Z

<AccordionGroup>

<Accordion title="Zero Knowledge Proofs">
A cryptographic method that allows one party to prove to another that a statement is true without revealing any additional information.
</Accordion>

</AccordionGroup>
